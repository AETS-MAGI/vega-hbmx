## 実験全体の判定ルール

### 成功判定
- HTTP 200 で応答が返る
- `result.json` 上で `status="ok"`
- `eval_count` が記録される
- runner crash が journal に出ない

### 失敗判定
- HTTP 500
- `status="error"`
- journal に `SIGSEGV`, `EOF`, `computeBatch`, `ggml_backend_sched_graph_compute_async` などが出る

### 必ず残す項目
- 実行時刻
- 使用スクリプト
- `OLLAMA_HOST`
- `MODEL`
- `NUM_PREDICT`
- `NUM_GPU`
- 生成された `run_id`
- 一言メモ（成功/失敗、気づき）

## 実験1: 同一条件での再現確認をもう1セット

### 目的
「たまたま1回通った/落ちた」を避け、**再現性がある**ことを補強する。

### 使用コード
- `vega-loop_qwen_rocm.py`

### 実施条件
- `MODEL=qwen3.5:2b`
- `NUM_PREDICT=512`
- `NUM_GPU=0,1,2,-1`
- `EPOCHS=1`
- ROCm と Vulkan で同一 prompt

### 実行コマンド
```bash
cd /home/limonene/ROCm-project/vega-investigate
source .venv-test/bin/activate

# ROCm
OLLAMA_HOST=http://127.0.0.1:11435 EPOCHS=1 NUM_PREDICT=512 NUM_GPU=0,1,2,-1 python vega-loop_qwen_rocm.py

# Vulkan
OLLAMA_HOST=http://127.0.0.1:11434 EPOCHS=1 NUM_PREDICT=512 NUM_GPU=0,1,2,-1 python vega-loop_qwen_rocm.py
```

### 人間の作業
- 実行前に対象サービスが正しい backend で立っているか確認
- 実行後に `run_id` を控える

### AIの作業
- 出力された `run_id` ごとの `result.json` を比較
- `summary_by_num_gpu` を1表にまとめる

### この実験で欲しい結論
- ROCm は再度全条件 `ok` か
- Vulkan は再度 `num_gpu=0` のみ `ok` か
- 既存結果と同じ傾向が再現したか

### 目的
`HSA_OVERRIDE_GFX_VERSION=9.0.0` の寄与を明示し、**override-assisted** であることを整理する。

### 使用コード
- `vega-loop_qwen_rocm.py`

### 実施条件
- ROCm endpoint
- `NUM_GPU=0` のみでよい
- override あり / なし の2条件

### 実行時の考え方
- ここはサービス起動条件の差が結果に直結するため、人間が状態を把握しながら進める
