Experiment History 実験の流れ

What we tried, what happened, and how the investigation advanced 何を行い、どういう結果が出て、どう進めたかを時系列で整理

Starting from uncertainty around legacy Vega support, the workflow moved step-by-step from environment control, to semantic validation, to matched backend tests, and finally to reproducible failure localization. 旧世代Vegaの実行可否が不明な状態から、環境統制→意味解釈の検証→バックエンド比較実験→再現可能な故障局在化へと段階的に進めた。

1) Problem framing and baseline setup 1) 問題設定と基盤整備

  • Did:実施: Defined the question as “support policy vs actual executability,” and fixed hardware/model/workload conditions.「公式サポート状況」と「実行可能性」を分離して問いを定義し、ハードウェア・モデル・負荷条件を固定した。
  • Result:結果: A consistent testbed was prepared so that backend differences could be observed without confounding factors.交絡要因を抑えた比較基盤ができ、バックエンド差を観測できる状態になった。
  • Next:次: Checked each execution gate from build filters to runtime initialization.ビルドフィルタから実行時初期化まで、各レイヤの通過条件を点検した。

2) Gate analysis across build and runtime layers 2) ビルド〜実行時レイヤのゲート解析

  • Did:実施: Inspected support matrix, CMake target filters, shipped ROCm artifacts, and initialization logs for gfx900.gfx900について、サポート表・CMakeターゲット制約・配布アーティファクト・初期化ログを横断確認した。
  • Result:結果: Default build policy excluded gfx900, but residual artifacts and runtime recognition still allowed execution paths in this environment.デフォルトビルドではgfx900が外れていた一方、この環境では残存アーティファクトとランタイム認識により実行経路が残っていた。
  • Next:次: Validated API semantics to avoid misinterpreting test parameters.実験パラメータ誤読を防ぐため、API意味解釈を検証した。

3) Semantic trace of num_gpu 3) num_gpu の意味トレース

  • Did:実施: Traced num_gpu end-to-end: Python client → server assignment → runner bridge → llama.cpp parameter.num_gpuを Pythonクライアント→サーバ割当→ランナーブリッジ→llama.cpp パラメータまで追跡した。
  • Result:結果: num_gpu was confirmed as “number of offloaded layers,” not number of GPU devices.num_gpuは「GPU台数」ではなく「GPUへオフロードする層数」であることを確認した。
  • Next:次: Ran matched ROCm and Vulkan experiments under the same model/prompt settings.同一モデル・同一プロンプト条件でROCmとVulkanの比較実験に進んだ。

4) Matched backend comparison experiments 4) バックエンド一致条件での比較実験

  • Did:実施: Executed the same workload with num_gpu = 0, 1, 2, -1 on both backends.両バックエンドで num_gpu = 0, 1, 2, -1 を同一ワークロード条件で実行した。
  • Result:結果: ROCm/HIP completed all tested settings, while Vulkan succeeded only at num_gpu=0 and failed with SIGSEGV when GPU offload was active.ROCm/HIPは全設定で完走した一方、Vulkanはnum_gpu=0のみ成功し、GPUオフロード有効時にSIGSEGVで失敗した。
  • Next:次: Repeated runs and aggregated logs to verify reproducibility and timing patterns.再現性とタイミング特性を確認するため、反復実行とログ集計を行った。

5) Reproduction and quantitative trend extraction 5) 再現確認と定量傾向の抽出

  • Did:実施: Repeated runs, then summarized by backend × num_gpu, including crash-time statistics.反復実行後に backend × num_gpu で集計し、クラッシュまでの時間統計も算出した。
  • Result:結果: A stable split emerged: ROCm remained successful in tested settings; Vulkan failures for num_gpu≥1 showed consistent time-to-crash bands.ROCm成功・Vulkan失敗（num_gpu≥1）という分離が安定して現れ、クラッシュ時間帯にも一貫性が見られた。
  • Next:次: Localized the failure stage in runtime stack traces.ランタイムスタックトレース上で故障段階の局在化を行った。

6) Failure localization and final interpretation 6) 故障局在化と最終解釈

  • Did:実施: Analyzed journal/stack traces around runner startup and compute execution.ランナー起動前後と計算実行部の journal・スタックトレースを解析した。
  • Result:結果: The crash occurred in Vulkan compute scheduling after successful loading, indicating a backend compute-path issue rather than universal Vega unusability.クラッシュはロード成功後のVulkan計算スケジューリング段で発生し、Vega全面不可ではなくバックエンド計算経路の問題と判断できた。
  • Next:次: Expand model/workload coverage and stress length while preserving the same evidence-first logging pipeline.同じエビデンス重視ログ基盤を維持しつつ、モデル種類・負荷長の拡張検証へ進む。
Key takeaway: The investigation progressed from assumptions to reproducible evidence, and the final claim is bounded to tested conditions. 要点: 仮説ベースではなく再現可能な証拠へ段階的に進め、最終主張は検証条件内に限定している。
