Background & Motivation[cite: 5]

  • Official support matrices not listed gfx900 in ROCm 7.2, but real systems can still execute through residual code and artifact paths.[cite: 6]
  • For students and small labs, the key question is whether execution is possible, stable, and reproducible[cite: 7]
  • We built a matched dual-backend testbed on the same Vega device to separate support policy, backend choice, and runtime behavior.[cite: 8]
  • Goal: explain why "unsupported" and "unusable" are not the same statement.[cite: 9]
2

Problem Statement[cite: 10]

  • Legacy Vega GPUs are often treated as categorically unsuitable for modern ROCm workloads[cite: 11]
  • Success or failure may be decided at different layers: distribution presets, build targets, init validation, backend macros, or runtime compute paths.[cite: 12]
  • We ask: is instability caused by Vega itself, or by a specific backend path under matched conditions?[cite: 13]

C. Related Literature (cited, not reproduced in this repo)[cite: 35]

  • LLNL differential testing (arXiv:2410.09172): 600K auto-generated kernels showed FP32 divergence "9% between NVIDIA V100 and AMD M1250. Root causes vendor-specific math libraries, compiler flags, HIPIFY artifacts. Motivates our backend-aware validation approach[cite: 37]
  • gem5 Vega modeling (Umv. Wisconen-Madison): GPUFS mode runs real Linux kernel unmodified amokid driver.[cite: 38] Validated against physical Vega 56 with avg 6% error. Establishes Vega as academically credible.[cite: 39]
  • NOCM SDMA workarounds (AMD docs LUMI reports): HSA_ENABLE_SOMA bypasses faulty DMA with Blit kernel fallback (80% bandwidth).[cite: 40] HSA OVERRIDE GFX VERSION enables unsupported GPL execution the override used in our experiments. These are external references.[cite: 41]
  • All claims in this poster are based solely on repo-internal evistence [Sections 3A, 38, 51[cite: 42]

Engineering Challenges[cite: 43]

  • gfx900 excluded by official matrices and default build filters, yet residual code and artifacts still retain g900 handling[cite: 44]
  • num gpu' is easily misread as GPU count: our code trace confirms it means offloaded layers.[cite: 45]
  • The same Vega GPU can succeed on ROCm/HIP and fail on Vulkan under the same model and prompt.[cite: 46]
  • Diagnosis must be layer-by-layer and evidence-driven, not reduced to "legacy GPU bad".[cite: 47]

Why gfx900 Still Works Sometimes[cite: 49]

ROCm changelog says "no longer built by default" rather than "forbidden."[cite: 50] in the local install, gfx900 hisaco artifacts and libggmi-hip.so strings were verified present.[cite: 51]

Interpretation[cite: 52] unsupported unguaranteed and untested, not necessarily impassible[cite: 53]

Limitations & Future Work[cite: 54]

Limitations: Matched comparison used short fixed runs (1 epoch, 512 tokens) with a single model (qwen3.5:2b).[cite: 55] Vulkan and ROCm Ollama versions differ (0.17.4 vs 0:17.5) tiny lama behaved differently from qwen3.5 on Vulkan, indicating model-dependent failure conditions.[cite: 56] Results should not be generalized beyond this tested scope without further verification.[cite: 57]

Future work:[cite: 58] Multi-epoch stress tests, anditional models (lim-jp, phi-3), ROCm version comparison (6.x.vs 7.x), and GGML CUDA NO PEER COPY experiments to isolate offload-path failure mechanisms.[cite: 59] Reproducibility infrastructure workaround matrix with 15 documented cases) is available in the repo for community use[cite: 60]

Takeaway & Key Messages[cite: 61]

✅ Support Status Execution Reality[cite: 62]

Official exclusion does not imply impossibility. Gate matrix shows 5 of 6 layers allow gfx500 execution.[cite: 63] Residual code paths and shipped artifacts make partial execution possible.[cite: 64]

💡 Backend Choice Dominates[cite: 65]

On the same Vega device with qwen3.5:26, ROCm/HIP passed all num_gpu conditions within tested scope.[cite: 66] Vulkan failed when num gpu>1 (GPU offload active). Failure is backend-specific, not architecture-wide. Model dependency (tinyllarna vs qwen) exists[cite: 67]

🔬 Reproducibility Before Claims[cite: 68]

All claims backed by specific run ids and code traces.[cite: 69] Key runs run 20260307 012643 ... workaround matrix (15 cases) in repo.[cite: 70]
